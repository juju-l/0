# 问题完整解析与修复方案

## 一、先搞懂 01007 是什么

`Code:01007 no privileges were granted` **不是报错，是警告**，含义：
你执行的这条`GRANT`没有新增任何权限，要么权限早已存在，要么当前执行用户**没有对应权限的转授权 \(WITH GRANT OPTION\)**，本次授权操作无变更。

你现在的场景是：

1. 超级用户`postgres`执行：

```sql
GRANT ALL PRIVILEGES ON DATABASE postgres TO "iam-adm@my-project.iam" WITH GRANT OPTION;
```

这条语句本身执行成功，赋予了`iam-adm`数据库级权限 \+ 转授权。
2\. 但你切到`iam-adm`去给其他 IAM 用户授权时，报出 3 条 01007 警告：

- `tstdb`（数据库级权限授权失败）

- `public`（schema 模式权限授权失败）

## 二、核心根因（2 个关键问题）

### 1\. 权限隔离：DATABASE 和 SCHEMA 是两类完全独立对象

`WITH GRANT OPTION`**只对本次授权的对象类型生效**：

- 你只给了`iam-adm` **DATABASE 数据库** 的转授权；

- `public`是**SCHEMA（模式）** 对象，你没单独给`iam-adm`授予`public` schema 权限 \+ 转授权；
因此`iam-adm`无法转授 schema 权限，执行`GRANT ... ON SCHEMA public`直接触发 01007 警告。

### 2\. Cloud SQL IAM 账号特殊限制

GCP Cloud SQL PostgreSQL 的 IAM 用户存在两层约束：

1. 数据库超级用户`postgres`是唯一内置 superuser；

2. IAM 账号即便拿到`DATABASE ALL WITH GRANT OPTION`，**不会自动继承 schema、表、序列等子对象的转授权限**，必须单独给 schema 授予带转授选项的权限。

### 3\. 补充：如果目标用户已有权限也会出 01007

若你重复执行相同 GRANT，数据库检测到权限已存在，不会重复授予，同样返回`no privileges were granted`警告，属于正常提示。

## 三、分步修复方案

### 步骤 1：用 postgres 超级用户补齐 schema 转授权限

连接数据库用默认`postgres`账号执行，给管理员 IAM 账号同时授予数据库、public 模式权限 \+ 转授权：

```sql
-- 1. 数据库级权限（你原有语句保留）
GRANT ALL PRIVILEGES ON DATABASE postgres TO "iam-adm@my-project.iam" WITH GRANT OPTION;

-- 2. 新增：public模式权限 + 转授权（解决public警告）
GRANT ALL PRIVILEGES ON SCHEMA public TO "iam-adm@my-project.iam" WITH GRANT OPTION;

-- 3. 可选：未来新建表/序列自动拥有转授能力（推荐）
ALTER DEFAULT PRIVILEGES IN SCHEMA public GRANT ALL TO "iam-adm@my-project.iam" WITH GRANT OPTION;
```

### 步骤 2：验证管理员账号授权能力

切换登录`iam-adm@my-project.iam`，再执行给其他 IAM 用户授权语句，不再出现 01007 警告：

```sql
-- 数据库级 CONNECT / CREATE 权限
GRANT CREATE, CONNECT ON DATABASE postgres TO "other-user@my-project.iam";

-- public模式访问权限
GRANT USAGE, SELECT, INSERT, UPDATE, DELETE ON SCHEMA public TO "other-user@my-project.iam";
```

## 四、排查命令（确认当前权限）

登录 postgres 超级用户，查询管理员账号持有的可转授权限：

```sql
-- 查看数据库权限与grant选项
SELECT datname, grantee, privilege_type, grant_option
FROM pg_database d
JOIN pg_database_privs dp ON d.oid = dp.objid
WHERE grantee = 'iam-adm@my-project.iam';

-- 查看schema权限与grant选项
SELECT nspname, grantee, privilege_type, grant_option
FROM pg_namespace n
JOIN pg_namespace_privs np ON n.oid = np.objid
WHERE grantee = 'iam-adm@my-project.iam';
```

查询结果里`grant_option`列必须为`true`，才能转授对应权限给其他人。

## 五、补充说明

1. 01007 只是警告，不阻断 SQL 执行，但代表授权逻辑未生效；

2. `WITH GRANT OPTION`不能授予给`PUBLIC`，只能给单个用户 / 角色；

3. 如果你只需要授予`CONNECT/CREATE`数据库权限，不需要`ALL`，可以精准缩小权限范围：

```sql
GRANT CREATE, CONNECT ON DATABASE postgres TO "iam-adm@my-project.iam" WITH GRANT OPTION;
```

4. Cloud SQL 不允许 IAM 账号变成真正 superuser，所有跨对象授权都需要`postgres`预先分配带转授选项的权限。
